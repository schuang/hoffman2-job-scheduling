“hello world” job submission¶

In this tutorial, we will create a simple job script, submit it, wait for it to complete, inspect its output files and study its behaviors.

Note

A job script is a standard shell script with additional lines containing job parameters. Some people might call it a “command file” or “job file” but really it is just a shell script. All of your shell script programming knowledge applies!

Demo: Submitting the hello-world job

Create a job script¶

Log in Hoffman2 Cluster via a SSH terminal.

Create and enter a directory (folder) tutorial-job-sched in your home directory (so the exercises do not mess up your other files and directories) by typing these commands in a terminal:

mkdir tutorial-job-sched
cd tutorial-job-sched

Using a text editor to create the following job script [download]. Then save this file as hello-world.sh (under the tutorial-job-sched directory).

#!/bin/bash
#$ -l h_data=1G
#$ -l h_rt=1:00:00
#$ -cwd

echo "hello world!"
date

Submit the job¶

From the directory tutorial-job-sched, submit the job by the command:

qsub hello-world.sh

On the screen, you will see a message, similar to:

Your job 3455432 (""hello-world.sh"") has been submitted

The long integer in the message above is the job number (also called “job ID”).

Monitor job status¶

After submission, you can check the job status by the command:

qstat -u $USER

You should see an output similar to:

job-ID   prior  name         user    state submit/start at      queue   jclass slots ja-task-ID
--------------------------------------------------------------------------------------------
3455432 0.00000 hello-worl   sch      qw   07/17/2022 16:36:16                   1

If you have successfully submitted the job, but qstate -u $USER shows nothing, the job may have completed (or aborted).

In the above output, qw means the job is still pending in the queue, not started yet.

At some point, the job state will change from qw to r (running) and runs to completion.

The output files¶

After the job has completed, there are two files in the directory ~/tutorial-job-sched (recall that we used the job option -cwd to run in the “current” directory.):

cd ~/tutorial-job-sched
ls -l

You will see output files similar to the followings, among other things:

hello-world.sh.e3455432
hello-world.sh.o3455432

  • The file name ending .e3455432 is from the standard error.

  • The file name ending .o3455432 is from the standard output.

Unlike running a program interactively in a terminal, there is no “screen” attached to a batch job. All of outputs from from standard output and standard error are written into these two files, respectively.

What does it mean?¶

The first few lines starting with # specify the job parameters:

Title¶

Job parameters

Meaning

#!/bin/bash

This job script is a bash script

#$ -l h_data=1G

The job is limited to 2GB of virtual memory at runtime

#$ -l h_rt=1:00:00

The job’s runtime limit is 1 hour (HH:MM:SS format)

#$ -cwd

When the job starts, it will start from the current directory

Note

If you know Linux shell scripting, you will notice that the lines starting with # (except for #!, the shebang line) are comments a shell script point of view. However, when this shell script is submitted to the job scheduler, these #$ lines will be recognized as job parameters by the scheduler.

We did not specify a job name, so by default the script’s file name is used as the job name. Also, the standard output and standard error filenames also contain the job name.

We can specify the job name by the -N option:

#$ -N myjob

Also, all of the -l options may be combined into one line (order does not matter, but space is not permitted):

#$ -l h_rt=1:00:00,h_data=1G

If desired, we could combine the standard output and standard error into one file by the -j option:

#$ -j y

The resulting job script is [download]:

#!/bin/bash
#$ -l h_data=1G,h_rt=1:00:00
#$ -cwd
#$ -j y
#$ -N myjob

echo "hello world!"
date

This job will produce only one output file, e.g. myjob.o3478770:

$ cat myjob.o3478770
hello world!
Mon Jul 18 15:42:10 PDT 2022

If other standard output/error file names are preferred, they can be specified by the -e (standard error) and -o (standard output) options. To distinguish the output files from different job ID, we could utilize the runtime variable $JOB_ID as follows:

#$ -o output.$JOB_ID

The resulting job script is [download]:

#!/bin/bash
#$ -l h_data=1G,h_rt=1:00:00
#$ -cwd
#$ -j y
#$ -N myjob
#$ -o output.$JOB_NAME.$JOB_ID

echo "hello world!"
date

Note

The following runtime variables are supported in the -e and -o options:

  • $HOME: home directory

  • $USER: user ID of job owner

  • $JOB_ID: job ID

  • $JOB_NAME job name (see -N option)

  • $HOSTNAME name of the compute node

  • $TASK_ID array job task index number (only for job arrays)

Quiz¶

  1. What is/are wrong in this job script?

#!/bin/bash
#$ -l h_data = 2G
#$ -l h_rt = 1:00:00
#$ -N myjob
#$ -o stdout.$JOBID
#$ -e stderr.$JOBID
#$ -cwd

echo "hello world!"
Answer

There are two problems:

  1. The runtime variable $JOBID is misspelled (therefore undefined). It should be $JOB_ID instead.

  2. Space is not allowed in specifying the job parameters. It should be h_data=2GB and h_rt=1:00:00, respectively.
