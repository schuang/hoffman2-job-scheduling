“hello world” job script¶

In this tutorial, we will create a simple job script, submit it, wait for it to complete, and inspect its output files.

Note

A job script is just a standard shell script with additional lines containing job parameters. Some people might call it a “command file” or “job file” but really it is just a shell script. All of your shell script programming knowledge applies!

Demo: Submitting the hello-world job

Create the job script¶

Log in Hoffman2 Cluster via a SSH terminal.

Create and enter a directory (folder) tutorial-job-sched in your home directory:

mkdir tutorial-job-sched
cd tutorial-job-sched

Using a text editor, create the following job script [download]. Then save this file as hello-world.sh (under the tutorial-job-sched directory).

#!/bin/bash
#$ -l h_data=2G
#$ -l h_rt=1:00:00
#$ -N myjob
#$ -o stdout.$JOB_ID
#$ -e stderr.$JOB_ID
#$ -cwd

echo "hello world!"
hostname
date
uname -a

What does it mean?¶

The first few lines starting with # specify the job parameters:

Title¶

Job parameters

Meaning

#!/bin/bash

This job script is a bash script

#$ -l h_data=2G

At runtime, the job is limited to 2GB of virtual memory

#$ -l h_rt=1:00:00

The job’s runtime limit is 1 hour.

#$ -N myjob

The job’s name (can be any string without spaces)

#$ -cwd

When the job starts, it will start from the current directory

#$ -o stdout.$JOB_ID

The runtime standard output will be saved in this file

#$ -e stderr.$JOB_ID

The runtime standard error will be saved in this file

Note

If you know Linux shell scripting, you will notice that the lines starting with # (except for #!, the shebang line) are just comments from the standard point of a shell script. Indeed, when a job script is run as a normal shell script, these # lines are just comments, as expected. However, when this shell script is submitted to the job scheduler, these lines will be recognized as job parameters by the scheduler, and the specified computing resources will be requested accordingly.

Submit the job¶

From the directory tutorial-job-sched, submit the job by the following command:

qsub hello-world.sh

On the scree, you will see a message, similar to:

Your job 3455432 ("myjob") has been submitted

The number is the job ID. The name in the quotes is the job name specified by the -N option mentioned above.

Monitor job status¶

After submission, you can check the job status by the command:

qstat -u $USER

You should see an output similar to:

job-ID   prior  name    user    state submit/start at      queue   jclass slots ja-task-ID
--------------------------------------------------------------------------------------------
3455432 0.00000 myjob   sch      qw   07/17/2022 16:36:16                   1

If you have successfully submitted the job, but qstate -u $USER shows nothing, the job may have completed.

In the above output, qw means the job is still pending in the queue, not started yet.

At some point, the job state will change from qw to r (running). After the job has completed, you won’t see the job in the qstat output.

Inspect the job output¶

After the job has completed, there will be at least two files in the directory ~/tutorial-job-sched (recall that we used the job option -cwd to run in the “current” directory.):

cd ~/tutorial-job-sched
ls -l

You should see an output similar to the following, among other things:

-rw-r--r--. 1 sch sch   0 Jul 17 16:36 stderr.3455432
-rw-r--r--. 1 sch sch 149 Jul 17 16:21 stdout.3455432

Summary¶

In this exercise, we have performed the following tasks:

  • Create a simple job script, which is a shell script with additional “comment” lines specifying the job parameters

  • Submit the job

  • Check (or monitor) job status

  • Inspect the job output files

We have also learned that:

  • Two ways to specify job parameters

    • In the job script, i.e. the #$ lines

    • from the command line

  • standard output and standard error can be combined

  • Job parameters can contain special runtime variables, such as $JOB_ID

Quiz¶

  1. What is wrong with this job script?

#!/bin/bash
#$ -l h_data = 2G
#$ -l h_rt = 1:00:00
#$ -N myjob
#$ -o stdout.$JOBID
#$ -e stderr.$JOBID
#$ -cwd

echo "hello world!"
Answer

  1. The runtime variable JOBID is undefined. It should be $JOB_ID instead.

  2. Space is not allowed in specifying the job parameters. It should be h_data=2GB and h_rt=1:00:00, respectively.
